# Expo Vision Camera App 開発レポート

本レポートは、`react-native-vision-camera` を用いたExpoプロジェクトの初期構築から、Digest認証機能実装によるアップロード・ポーリング処理の正常動作までに得られた知見、発生したトラブルとその解決策、および今後の開発に役立つプロンプト例をまとめたものです。

## 1. 環境構築・立ち上げ時のポイント

### ネイティブ依存機能と EAS Build
*   **課題**: `react-native-vision-camera` など、ネイティブOSのハードウェア（カメラなど）に深くアクセスするライブラリは、標準の Expo Go アプリでは動作しません。
*   **解決**: プロジェクトの開始初期から、クラウドビルドサービスである **EAS (Expo Application Services) Build** を使用して、カスタムの開発用クライアント（Development Build）を作成する方針に切り替えました。
*   **初期インストールの詰まり**: `npx create-expo-app` 実行時にパッケージのインストールがハングアップする現象が発生しました。これは手動でプロジェクトフォルダに入り、再度 `npm install` および `npx expo install` を実行することで解決しました。

## 2. 実装中に発生した主要なトラブルと解決策

### ① Expo FileSystem のアップデートによる import エラー
*   **課題**: `expo-file-system` を用いて設定ファイル（`config.json`）を読み書きしようとした際、「`Method getInfoAsync imported from "expo-file-system" is deprecated.`」というエラーでクラッシュしました。
*   **解決**: 最新のExpo SDK環境において、従来のファイル操作APIが `legacy` モジュールに移行されていたためでした。インポート元を `expo-file-system/legacy` に修正することで正常に動作しました。スマホ内のローカルファイル永続化には今後もこのパス変更に注意が必要です。

### ② "Camera is closed" エラー（カメラのライフサイクル）
*   **課題**: 「撮像」ボタンを押すと即座に「Camera is closed」というエラーが発生し、写真が撮れませんでした。
*   **解決**: 撮影開始と同時に画面に「処理中...」のUIを表示するためにステータス（`isProcessing`）を `true` に切り替えていました。しかし、カメラコンポーネントの `isActive` 属性を `isActive={!isProcessing}` と設定していたため、**写真を撮り終える前にカメラ自体がオフライン状態になって強制切断される**という状態に陥っていました。撮影中およびその直後も `isActive={true}` を保持するように修正し、解決しました。

### ③ モバイルアプリによる意図しない HTTPS へのアップグレード
*   **課題**: Pythonで構築したモックサーバー（HTTP, ポート31000）に対して、アプリからリクエストを送信すると `Bad request version` というハンドシェイクエラーがサーバー側で発生しました。
*   **解決**: 最近のiOS/Androidはセキュリティ上、ローカルIPへの通信も含めて自動的に `https://` へ通信をアップグレードして送信してしまうことがあります。APIにリクエストを送る直前のURL生成処理で、強制的に `http://` に書き換える処理（ダウングレード）を実装して解決しました。

### ④ Digest 認証の厳密な仕様とサーバー側ライブラリのバグ
*   **課題**: ユーザー名、パスワード、Realm、Nonce 等を用いてMD5ハッシュを作成する Digest 認証（RFC 2617）を自前で実装しました。ハッシュ自体は規格通り100%正しく算出されていましたが、Pythonサーバー側の `flask-httpauth` ライブラリの内部検証処理で無条件に 401 (Unauthorized) 弾かれてしまっていました。
*   **解決**: サーバー側の汎用ライブラリが想定するヘッダーの厳密な記法（`qop` や `nc` のダブルクォートの有無、アルゴリズム指定など）の不一致が原因でした。アプリ側のハッシュ算出アルゴリズムは完璧であったため、サーバー側を「アプリが送信した生のAuthorizationヘッダーの定義に従って自力でMD5を突合する」ロジックに書き換えることで完全に突破しました。

## 3. 次回の開発をスムーズにするための「魔法のプロンプト」

今回の教訓を活かし、次回以降、似たようなプロジェクト（Expo + カメラ/ネイティブ機能 + 認証付きAPI）をAIに依頼する際は、最初のプロンプトに以下の条件を含めると、回り道をせずに最短で正常なアプリが完成します。

> **【次回以降の開発用 おすすめ追加プロンプト】**
> 
> *   **環境について**: 「`react-native-vision-camera` 等のネイティブ機能を使用するため、Expo Goではなく、EAS BuildによるDevelopment Buildを前提とした環境構築と `app.json` のプラグイン設定を行ってください。」
> *   **ファイル操作について**: 「ローカルファイルアクセスに `expo-file-system` を使用する場合、最新のExpo SDKに準拠し、非推奨エラーを避けるために `expo-file-system/legacy` からメソッドをimportしてください。」
> *   **カメラ制御について**: 「カメラの撮像処理（`takePhoto`等）を行う際、非同期通信中やローディング中であっても、キャプチャが完全に完了するまではカメラコンポーネントの `isActive` プロパティが `false` にならないように注意して実装してください（Camera is closed回避のため）。」
> *   **通信・Digest認証について**: 「Digest認証の Authorization ヘッダーを生成する際は、RFC 2617 の規格に厳格に準拠してください。特に `qop` や `nc` のダブルクォートの有無によるサーバー側のパースエラーに注意し、確実なMD5ハッシュ文字列を生成する関数を実装してください。また、開発環境では意図しないHTTPS通信への自動変更を防ぐため、URLを明示的に `http://` に固定する処理を入れてください。」

**総括**
ネイティブハードウェアの制御、ローカルファイルへの設定の永続化、そして複雑な暗号化通信（Digest認証）という、複数の非常に難易度の高いトピックが組み合わさったプロジェクトでしたが、ひとつひとつのログを解析することで無事にフルスクラッチでの結合テストに成功しました。お疲れ様でした！
